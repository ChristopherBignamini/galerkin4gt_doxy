Mass Matrix Discretization

The simplest example that we can consider is the case of a mass matrix assembly for a scalar field. We consider a 3D domain, with the structured grid dimension specified via command line. The discretization of a scalar field in this framework will lead to a 4D tensor: three indices identifying an element and one index identifying the degree of freedom inside the element. A matrix (local to the element) is represented as a 5D tensors, with 2 indices for the degrees of freedom inside the element.

The memory layout plays an important role in HPC, and for this reason it is exposed by the GridTools API. In this framework we will use the gdl::layout_tt alias which is choosing the optimal memory layout for us, and we just pass in the dimensionality of the arrays we want to define, as shown in the snippet below. examples/mass.cpp definitions

The storage_info metafunction generates a type which must be attached to the storage, and which is giving a "shape" to a storage (i.e. how many dimensions, with which memory layout)

The type definitions reported above set the discretization that we want to use for the solution (i.e. "fe") and for the geometric mapping (i.e. "geo_map"). The three template arguments of gdl::reference_element represent in order the discretization order, the type of basis function, and the shape of the element.

The type of integration employed is also defined by the type gdl::cubature, getting an integration order and a shape. Note that the integration order should integrate a composition of the basis functions with the geometric map, which might result e.g. in a hight order polynomial. an integration order which integrates inexactly these quantities can be used in such cases.

Notice that the discretization and quadrature are not depending on each other. They get associated in the gdl::intrepid::geometry and gdl::intrepid::discretization, which are responsible to compute and keep the values of the local quantities in the quadrature points of the reference element. These objects are defined from a gdl::discretization and a gdl::cubature

In the following snippet the former types get instantiated examples/mass.cpp instantiation the gdl::intrepid::geometry::compute and gdl::intrepid::discretization::compute use enum types like Intrepid::OPERATOR_VALUE (from the Intrepid library) to compute the operators in the quadrature points of the reference element. So far we dealt with small storages, everything was local to the reference element. Now we define an assembler, which has the notion of the whole grid, together with the geometric map, we will use this assembler to store the grid and the map from the reference to the actual configurations. examples/mass.cpp as_definition Notice that this assembler does not know anything about the discretization of the solution: it contains only the geometric information.

We instantiate the assembler examples/mass.cpp as_instantiation

Now we instantiate the local mass matrix, as a 5D tensor. We use the GridTools API, so we first instantiate a storage_info, which gives the shape to the raw storage and can be shared among several storages, and then we pass the storage_info instance to the storage constructor examples/mass.cpp instantiation_mass We pass in the 5 dimensions of the storage to the storage_info constructor: the number of elements in the 3 spatial coordinates and twice the number of degreed of freedoms for the chosen discretization

The following lines are plain GridTools syntax to define the computation examples/mass.cpp args We give a name to each argument that will be used in the computation tree.

The core of the computation is defined and executed in the following lines examples/mass.cpp computation

in which we first define the computation tree. The information provided to the make_computation are processed in the compilation step to produce the actual algorithm to be run on the specific backend passed as template parameter. This step only defines the computation, which is then lazily executed in the subsequent steps.

In this example we used functors provided by the library, namely gdl::functors::update_jac, gdl::functors::det, gdl::functors::mass. These compute the jacobian matrix, its determinant, and compute the mass. Such functors can be defined directly at the user level, and represent the innermost part of the kernel being executed.

The first functor updates the jacobian matrix: functors/assembly_functors.hpp update_jac

The second functor computes the determinant of a matrix functors/assembly_functors.hpp det

The last functor computes the mass matrix in the actual reference frame functors/mass.hpp integration
